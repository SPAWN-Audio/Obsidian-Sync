## Claude Code Summaries
### Objects & Instances

A **class** is a blueprint. An **object** is something built from that blueprint. An **instance** is just a specific copy of that object living in your game.

Think of it like a cookie cutter (the class) and the actual cookies (instances). Every cookie came from the same cutter, but each one exists independently — you can put different amounts of frosting on each one without affecting the others.

```csharp
// This is the blueprint (the class/script)
public class Enemy : MonoBehaviour
{
    public int health = 100;
}

// Somewhere else, you can make instances of it:
Enemy goblin = new Enemy();
Enemy dragon = new Enemy();

goblin.health = 50;   // only the goblin is affected
dragon.health = 500;  // only the dragon is affected
```

In Unity, every time you drag a script onto a GameObject, or duplicate an object in the scene — you're creating a new **instance**. They all share the same blueprint but live their own independent lives.

### Cross Script Referencing

Sometimes Script A needs to talk to Script B. Maybe your `Player` script needs to tell the `ScoreManager` script to add points. This is **cross script referencing** — getting a handle on another script so you can read or call things on it.

The most common way:

```csharp
// On your Player script, grab a reference to ScoreManager
public ScoreManager scoreManager; // drag it in via Inspector

void PickupCoin()
{
    scoreManager.AddScore(10); // now you can call its functions
}
```

You can also find it in code without the Inspector:

```csharp
// Find it by looking for it on a specific GameObject
ScoreManager scoreManager = GameObject.Find("ScoreManager").GetComponent<ScoreManager>();
```

Think of it like getting someone's phone number — once you have the reference, you can call them (their functions) or ask them things (their variables) whenever you need.

> ⚠️ **Tip:** Prefer dragging references in via the Inspector where possible. `GameObject.Find()` is slow and fragile — if the object gets renamed, it breaks.

### Check for Null

**Null** means "nothing is here." A variable that points to nothing is null. If you try to _use_ something that's null — call a function on it, read a value from it — your game crashes with a `NullReferenceException`.

Checking for null is just asking _"does this thing actually exist before I try to use it?"_

```csharp
// Dangerous — if enemy is null, this crashes
enemy.TakeDamage(10);

// Safe — check first
if (enemy != null)
{
    enemy.TakeDamage(10);
}
```

A shorter modern way to do the same thing is the **null-conditional operator**:

```csharp
enemy?.TakeDamage(10); // the ? means "only do this if enemy is not null"
```

Common situations where null bites you:

- You forgot to drag something into an Inspector slot
- The object you're looking for was destroyed
- A `GetComponent<>()` call found nothing (the component doesn't exist on that object)

Whenever you're unsure if a reference is valid — check it. It's one of the most common sources of bugs in Unity.

### Audio

Unity plays audio through two main pieces: an **AudioClip** (the actual sound file) and an **AudioSource** (the speaker that plays it).

Think of an `AudioClip` as a CD, and the `AudioSource` as the CD player. You load a disc into the player and hit play.

**Basic setup — attach an AudioSource to your GameObject:**

```csharp
public AudioSource audioSource;
public AudioClip explosionSound;

void Explode()
{
    audioSource.PlayOneShot(explosionSound);
}
```

`PlayOneShot` is great for sound effects — it plays the clip once and doesn't interrupt anything else already playing.

**Playing a clip at a location in the world** (no AudioSource needed):

```csharp
// Plays a sound at a position then auto-cleans up
AudioSource.PlayClipAtPoint(explosionSound, transform.position);
```

**For background music**, just tick **Loop** on the AudioSource in the Inspector and call:

```csharp
audioSource.Play();
```

**Quick reference for common AudioSource controls:**

```csharp
audioSource.Play();         // start playing
audioSource.Stop();         // stop completely
audioSource.Pause();        // pause
audioSource.volume = 0.5f;  // half volume (0 to 1)
audioSource.loop = true;    // loop it
```

> 🔊 **Tip:** For music use `Play()` with loop on. For sound effects use `PlayOneShot()` so multiple overlapping sounds (like gunshots) can all play at once without cutting each other off.

## Cross Script Referencing 
Cross-script referencing is the process of one script accessing variables, functions, or components
of another script.
### Benefits:
- Promotes modularity and reusability of code.
- Allows multiple scripts to collaborate and share information.
- Essential for interactions between different game elements and functionalities.

The ideal scenario is that we have one script, that does one job, i.e. I shouldn't see movement code inside of a Health script. This is the first part of our S.O.L.I.D principles, S is Single Responsibility.

In this example we have two objects in our scene one with a player health script attach and the other with enemy attack.

To get a reference to the player health script we declare public variables or fields in one script.

Then assign the PlayerHealth reference to the variable of that type through the Unity Inspector (drag and drop the object the script is attached to in the inspector). 

Using this variable and our `.` we can access any public functions/variables through our reference. Think of the `.` as I am looking inside of this script or container.

## Check For Null
When it comes to using references to other code, the most common error code you will get is a null reference error.
This is basically Unity saying you have asked to use a reference to a script, however you haven't provided a
reference, so it attempts to access nothing and throws the null reference error in the process.

So best practice when using references is to check to see the reference exists before you try to access it, we call this
null checking. You want to try and resolve null errors as soon as possible as leaving them will cause other errors in
your code at runtime.

## Audio
Audio is a crucial aspect of creating immersive and engaging games.
Unity provides robust tools for managing and playing audio.
#### Benefits:
- Atmosphere: Set the mood and create an immersive environment.
- Feedback: Provide auditory feedback for actions and events.
- Storytelling: Enhance storytelling with voiceovers and sound effects.
There are two major elements for audio.
#### AudioSource:
- Represents a sound emitter attached to a GameObject. 
- Plays audio clips, controlling volume, pitch, and spatial effects.
#### AudioListener:
- Represents the player's ears in the scene.
- Captures and processes audio, enabling spatial audio effects.

Unity can handle most file types, to import audio simply drag and drop them into your assets folder.

[Unity - Manual: Audio overview](https://docs.unity3d.com/Manual/AudioOverview.html)

### Audio Examples
### Use Cases:
#### Background Music:
- Play music that suits the game's mood and setting.
#### Sound Effects:
- Trigger sounds for actions, interactions, or events.
#### Voiceovers:
- Deliver narrative or character dialogue through audio.
#### Ambient Sounds:
- Add realism with environmental sounds (wind, water, etc.).
